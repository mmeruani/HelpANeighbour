# Help A Neighbour

`Help A Neighbour` — кроссплатформенное клиент-серверное мобильное приложение для организации бытовой взаимопомощи внутри локальных сообществ. Проект разработан на Flutter с использованием Firebase и ориентирован на жителей домов, подъездов, жилых комплексов, СНТ, коттеджных поселков, улиц и других территориально близких объединений.

Приложение переводит соседские просьбы из неструктурированных чатов и личных сообщений в формат карточек запросов: с категориями, откликами, статусами выполнения, уведомлениями, историей услуг, отзывами и рейтингом пользователей.

## Назначение

Цель проекта — предоставить жителям локальных сообществ удобный инструмент для поиска и оказания бытовой помощи рядом с домом. Приложение подходит для ситуаций, когда проще обратиться к людям поблизости, чем искать исполнителя на крупной коммерческой платформе.

Примеры задач:

- купить продукты или забрать заказ;
- выгулять или покормить питомца;
- помочь с документами, распечаткой или сканированием;
- донести тяжелые вещи или помочь с переездом;
- выполнить мелкий бытовой ремонт;
- сопроводить ребенка, пожилого человека или соседа;
- одолжить инструмент или помочь с небольшой бытовой просьбой.

Ключевая особенность приложения — закрытый локальный формат. Пользователь видит запросы только внутри тех сообществ, в которых состоит, а вступление в сообщество происходит по коду или ссылке приглашения.

## Основные возможности

### Регистрация и вход

- регистрация пользователя с указанием имени, телефона, электронной почты и пароля;
- вход по электронной почте или номеру телефона;
- проверка корректности введенных данных;
- понятная обработка ошибок авторизации, включая неверный пароль;
- выход из аккаунта с подтверждением.

### Профиль пользователя

- просмотр и редактирование имени, описания и аватара;
- хранение контактных данных владельца аккаунта;
- отображение рейтинга, количества отзывов и завершенных услуг;
- доступ к истории заказчика, истории исполнителя и завершенным услугам;
- просмотр публичной страницы отзывов;
- настройка уведомлений.

### Сообщества

- создание локального сообщества с названием, описанием и изображением;
- вступление в сообщество по коду или ссылке приглашения;
- просмотр списка сообществ пользователя;
- просмотр участников сообщества;
- поддержка ролей `creator` и `participant`;
- изменение данных сообщества создателем;
- обновление приглашения, выход из сообщества и удаление сообщества.

### Запросы на услуги

- создание запроса на бытовую услугу;
- указание названия, категории, описания, срочности, желаемого срока выполнения, адреса и контактных данных;
- выбор условий вознаграждения:
  - без оплаты;
  - по договоренности;
  - фиксированная сумма;
- просмотр ленты запросов внутри сообщества;
- поиск и фильтрация запросов по категории, срочности, статусу, сроку выполнения и вознаграждению;
- редактирование и отмена активного запроса.

### Сценарий исполнителя

- просмотр запросов внутри сообщества;
- открытие подробной карточки запроса;
- отправка отклика с комментарием;
- отмена собственного отклика;
- просмотр принятых запросов;
- отметка услуги как выполненной после фактического выполнения.

### Сценарий заказчика

- просмотр откликов на созданный запрос;
- сравнение откликнувшихся пользователей по имени, аватару, рейтингу и комментарию;
- выбор исполнителя;
- отслеживание статуса запроса;
- подтверждение завершения услуги;
- оставление оценки и текстового отзыва после выполнения.

### Отзывы и рейтинг

- один отзыв создается для одной завершенной услуги;
- рейтинг пользователя пересчитывается на основе полученных оценок;
- отзывы отображаются на публичной странице пользователя;
- рейтинг и количество отзывов помогают оценить надежность участника перед взаимодействием.

### Уведомления

В приложении создаются in-app уведомления о значимых событиях:

- новый отклик на запрос;
- выбор исполнителя;
- отметка услуги как выполненной;
- подтверждение завершения услуги;
- новый отзыв.

Страница уведомлений поддерживает состояние прочитано/не прочитано, а в интерфейсе отображается счетчик непрочитанных событий. Firebase Cloud Messaging используется как инфраструктурная основа для push-уведомлений.

## Жизненный цикл запроса

Запрос проходит через несколько статусов:

```text
active
  -> inProgress
  -> awaitingCustomerConfirmation
  -> completed
```

Также запрос может быть отменен:

```text
cancelled
```

Описание статусов:

- `active` — запрос опубликован и доступен для откликов;
- `inProgress` — заказчик выбрал исполнителя;
- `awaitingCustomerConfirmation` — исполнитель отметил услугу как выполненную;
- `completed` — заказчик подтвердил выполнение услуги;
- `cancelled` — запрос отменен или больше не актуален.

## Архитектура

Проект построен по feature-based послойной архитектуре. Код разделен на функциональные модули, а внутри каждого модуля выделены слои `data`, `domain` и `presentation`.

Типовая структура функционального модуля:

```text
lib/features/<feature>/
  data/
  domain/
  presentation/
```

Роли слоев:

- `presentation` — экраны, виджеты, контроллеры состояния и UI-состояния;
- `domain` — сущности, интерфейсы репозиториев и use cases;
- `data` — реализации репозиториев, API-классы и работа с Firebase, Firestore и Cloudinary.

Такой подход отделяет пользовательский интерфейс от деталей хранения данных. Экраны обращаются к контроллерам, контроллеры вызывают репозитории, а репозитории используют API-классы для конкретных операций с внешними сервисами.

## Безопасность данных

Доступ к данным ограничивается через `firestore.rules`. Правила учитывают авторизацию пользователя, владение документом, участие в сообществе и роль пользователя в конкретном запросе.

Примеры ограничений:

- пользователь может читать и изменять только свой приватный профиль;
- публичные данные профиля доступны авторизованным пользователям;
- запросы сообщества доступны участникам соответствующего сообщества;
- отклики видны исполнителю, заказчику или участникам сообщества;
- уведомления может изменять только получатель, причем только в части статуса прочтения;
- удаление критичных пользовательских данных через клиентские правила запрещено.

Файл правил:

```text
firestore.rules
```

## Локальный запуск

Клонировать репозиторий:

```bash
git clone https://github.com/mmeruani/HelpANeighbour.git
cd HelpANeighbour
```

Установить зависимости Flutter:

```bash
flutter pub get
```

Запустить приложение:

```bash
flutter run
```

Запуск на конкретной платформе:

```bash
flutter run -d android
flutter run -d ios
flutter run -d macos
```

## Настройка Firebase

В репозитории уже есть Firebase-конфигурация, использованная при разработке. Если приложение нужно подключить к другому Firebase-проекту, необходимо выполнить:

```bash
flutterfire configure
```

После этого должны быть обновлены:

```text
lib/firebase_options.dart
android/app/google-services.json
ios/Runner/GoogleService-Info.plist
firebase.json
```

Деплой правил Firestore:

```bash
firebase deploy --only firestore:rules
```

## Firebase Cloud Functions

Установить зависимости функций:

```bash
cd functions
npm install
```

Проверить синтаксис:

```bash
npm run lint
```

Задеплоить функции:

```bash
firebase deploy --only functions
```

Слой функций используется для серверной обработки уведомлений и операций, которые не должны полностью доверяться клиентскому приложению.

## Настройка Cloudinary

Аватары пользователей и изображения сообществ загружаются в Cloudinary через unsigned upload preset.

Конфигурация находится в:

```text
lib/core/config/env.dart
```

Ключевые поля:

```dart
static const cloudinaryCloudName = '...';
static const cloudinaryUnsignedUploadPreset = '...';
```

Для production-использования рекомендуется ограничить unsigned preset на стороне Cloudinary и не разрешать произвольные загрузки без ограничений.

## Особенности iOS

Для запуска iOS-версии с CocoaPods нужно открывать workspace, а не project-файл:

```bash
open ios/Runner.xcworkspace
```

Если pods не установлены:

```bash
cd ios
pod install
```

Если Xcode показывает ошибку `Framework 'Pods_Runner' not found`, обычно нужно открыть именно `Runner.xcworkspace` и заново установить pods.
